## 产品概述

> **UI 设计规范**（色彩、组件、MVP 页面线框）：见 [docs/UI-DESIGN.md](docs/UI-DESIGN.md)。

**底噪 (Background Noise)** 是一个专为独立音乐、摇滚乐及 Live House 爱好者打造的垂直记录与社交平台。它拒绝泛流量平台的噪音，通过“演出条目”化管理，为用户提供纯粹的现场记忆留存空间，并基于“共同记忆”激发高质量的陌生人社交。

**目标用户**：独立音乐/摇滚乐死忠粉、Live House 常客、乐评人。他们渴望记录每一场现场的躁动，寻找审美契合的同好，且反感泛社交平台的干扰和隐私压力。

**核心价值**：
- **垂直聚焦**：每一场演出都是独一无二的条目，内容高度聚拢，无无关信息干扰。
- **记忆共鸣**：通过“一句话记忆”标签，在茫茫人海中精准定位拥有相同感受的“在场者”。
- **纯粹社交**：基于音乐审美的陌生人社交，无现实社交压力，信噪比极高。

## 应用场景

- **演出复盘**：小王刚看完一场痛仰的演出，热血沸腾。他在“底噪”找到该场演出条目，上传了抓拍的视频，打出高分，并标记记忆为“那一刻我真的在跳水”。随后，他看到另一个用户的记忆标签是“最后的小号声绝了”，两人瞬间产生共鸣，开始在条目下深度交流。
- **演出考古**：资深乐迷老张想回忆 3 年前在某 Live House 看过的一场冷门乐队演出。通过搜索，他发现该条目下依然保留着当年几十个在场者的照片和文字，仿佛穿越回了那个夜晚。
- **寻找同好**：新入坑的小李想找人一起去下周的草莓音乐节。他在“想看”列表里发现了几个常听风格与自己高度重合的用户，通过查看他们的过往记录，确定了对方的审美取向，发起了组队邀请。
- **AI 灵感辅助**：用户小陈想发个朋友圈式的 repo 但词穷，他在“底噪”输入几个关键词“万能青年、音响炸裂、石家庄”，AI 自动生成了一段富有诗意和摇滚气息的文字，小陈稍作修改后发布。

## 功能需求

**核心功能**
- **演出条目百科 (Wiki 模式)**：
    - **事实信息**：唯一性定义为“单场演出（艺人+日期+场馆）”。
    - **视觉封面 (UGC)**：条目主图默认展示该条目下**点赞数最高**的用户现场照片。这种“众创封面”既规避了版权风险，又体现了现场最真实的氛围。
    - **版权物料**：严禁平台抓取官方海报，仅允许用户以“附件/参考”形式上传自拍的海报或门票。
- **现场记录 (Repo) 与“社交票根”验证 (POA)**：
    - **验证即勋章 (POA as Badge)**：
        - **现场碎片 (Live Fragment)**：用户上传含有 EXIF 信息（时间/地点匹配）的照片视频。系统自动生成一张极具设计感的**“电子票根”卡片**（含艺人、场馆、编号、验证戳）。
        - **记忆考古 (Memory Archeology)**：针对 1 年以上的老记录，用户上传任何凭证（旧票根、模糊照片或深度文字）。审核通过后生成**“复古记忆卡”**，并赋予“资深见证者”勋章。
        - **身份背书 (Identity Proof)**：KOL、乐评人、乐队亲友通过实名认证后，其所有记录自动获得“官方见证”标识。
    - **社交货币化**：这些验证后的“票根”或“记忆卡”支持一键分享至朋友圈，成为用户在摇滚圈的身份标识。
    - **个人记忆云盘**：作为用户的私有资产，永久保存其原始素材，支持按时间线/地图维度检索。
- **主观表达与排序**：
    - **三维 Vibe Check**：保留“乐队表现”、“音响效果”、“现场氛围”的个人记录。
    - **商业价值挖掘**：后端聚合匿名化的 Vibe 数据，为 B 端（乐队、场馆、主办）提供行业洞察报告。
    - **内容排序**：条目下 Repo 列表优先展示“已验证”且“高赞”的记录。
- **“一句话记忆”标记 (社交钩子)**：
    - 用户提交记录时，必须提炼一个核心关键词或短句。
    - 社交逻辑：遵循“好内容驱动社交”，不设强制交互，通过标签引发共鸣，引导用户自发交互。
- **音乐画像集成**：
    - 接入网易云音乐开放平台 API，同步用户音乐偏好标签。
    - 基于用户画像和 Vibe Check 数据，计算“审美重合度”。

**辅助功能**
- **AI 现场记录助手**：用户输入零散关键词 → AI 自动扩写为富有感染力的演出记录。
- **“共同记忆”社交**：系统自动识别共同参与过同一场演出的用户，并推荐“审美重合度”高的同好。
- **演出提醒**：关注的乐队有新演出时，系统自动推送通知。

## UI 布局

### 整体布局
采用移动端 App 经典的底部 Tab 导航结构：【现场】、【发现】、【记录】、【我的】。

### 【现场】（首页）
- 顶部：搜索框（搜索演出、乐队、场地）。
- 中部：正在发生/即将发生的演出卡片流。
- 底部：热门演出条目推荐。

### 【演出条目详情页】
- 顶部：演出海报（背景毛玻璃效果）+ 基础信息（艺人、时间、地点）。
- 中间：
    - 统计区：平均分（三个维度）、想看/看过的总人数。
    - **“记忆墙”**：横向滚动的“一句话记忆”标签云，点击可查看对应的详细记录。
- 下部：用户 Repo 列表，支持按“热门”、“最新”排序。

### 【记录发布页】
- 图片/视频上传区。
- 文本输入框。
- 三维评分滑块（1-5 星）。
- **“一句话记忆”输入框**（限 20 字内）。
- **AI 扩写按钮**：位于文本框下方。

### 【个人中心】
- 顶部：头像、昵称、音乐标签（来自集成平台）。
- 中部：演出统计（看过 X 场、去过 Y 个场地）。
- 下部：Tab 切换（我的记录、想看的演出、我的同好）。

## 用户使用流程

1. **查找演出**：用户通过首页推荐或搜索找到目标演出条目。
2. **记录现场**：
    - 点击“标记看过”。
    - 上传照片/视频，填写文字（可选 AI 扩写）。
    - 进行三维评分。
    - **填写“一句话记忆”**（核心步骤）。
3. **社交交互**：
    - 在条目详情页浏览他人的“一句话记忆”。
    - 发现共鸣，点赞记录或发起私聊/评论。
4. **管理画像**：在个人中心查看自己的演出历程，系统根据记录持续优化用户的音乐标签。

## AI 能力需求

| 能力类型 | 用途说明 | 应用位置 |
|---------|---------|---------|
| 文本生成 (LLM) | 将用户零散的关键词扩写为富有感染力的演出记录 (Repo) | 记录发布页 - AI 扩写按钮 |
| 文本摘要/关键词提取 | 自动从用户长篇记录中建议“一句话记忆”标签 | 记录发布页 - 自动填充建议 |
| 推荐算法 | 基于用户的演出历史 and 音乐标签推荐“审美同好” | 发现页 / 个人中心 |

## 技术说明

- **目标平台**：Web (PWA - Progressive Web App)。
- **开发框架**：React (TypeScript)。
- **后端服务**：独立后端 API (如 Express.js, NestJS) 或 Firebase。
- **数据存储**：云数据库 (如 MongoDB, PostgreSQL)。
- **外部依赖**：
    - 秀动/大麦数据：初期仅用于获取基础演出元数据。
    - 音乐平台 API：接入网易云音乐等公开 API 获取用户歌单/偏好。
- **安全性**：敏感 API Key 存储于服务端环境变量中。
