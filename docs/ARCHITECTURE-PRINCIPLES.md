# 系统架构原则（可扩展 / 鲁棒）

面向后续商业化、深度社交、多端与未知功能，实施时建议遵守以下边界；具体技术栈（NestJS、PostgreSQL 等）可再选型，**不绑定本文件**。

## 1. 分层与边界

- **领域层**：演出条目、Repo、用户、可见性、POA 状态、Vibe 维度、标签、关系（关注 / 同场）、推荐特征——**与 HTTP、推送通道、支付网关无关**。
- **应用服务层**：用例编排（发布记录、变更可见性、审核流转、推荐任务触发）。
- **适配层**：REST 或 GraphQL、移动端推送、第三方 OAuth、票务元数据同步、网易云 API、未来 IM / 支付 Webhook。
- **异步与增长**：搜索索引、推荐重算、报表、审计日志经**消息队列 + 消费者**，避免拖慢写路径。

## 2. 能力插件化预留

- **商业化**：会员、报告导出、B 端席位等建模为**权益包 + 策略引擎**（鉴权中间件或单独 Policy 服务），不在「发布 Repo」接口内硬编码 if-vip。
- **深度社交**：会话、关系链、举报、屏蔽、未成年人策略等**独立限界上下文**，与演出内容通过 `userId` / `showId` 关联，避免单模块巨石。
- **审核与合规**：POA 考古、UGC 封面、举报走统一 **审核任务 + 状态机**（可插人工队列、规则版本号）。

## 3. 数据、契约与配置

- **事件 + 快照**：可见性变更、验证结果等写审计事件，便于纠纷处理与推荐重放。
- **API 版本化**：对外 `/v1` 或 semver，避免小程序与双端升级互相阻塞。
- **配置驱动**：排序权重、推荐开关、实验桶走配置中心，支持灰度与回滚。

## 4. 鲁棒性

- **幂等**：上传完成回调、支付回调、审核回调使用幂等键。
- **降级**：推荐或 AI 失败时，核心路径仍可完成「保存记录」；AI 为可选增强。
- **配额与风控**：存储配额、写频率、敏感词、反爬在网关或统一中间层策略化。
