# 底噪 v0.2 — 记忆表达引导 & 共同记忆社交

> **版本**：v0.2（v0.1 MVP 基础上升级）
> **日期**：2026-05-23
> **关联**：v0.1 详见 [Product-Spec.md](./Product-Spec.md)、[FEATURE-BACKLOG.md](./FEATURE-BACKLOG.md)

---

## 一、概述

### 1.1 背景

v0.1（MVP）已完成基础链路：注册登录 → 演出条目 → 发布 Repo（记忆+评分） → 记忆墙 → 点赞+评论 → 个人时间线。

v0.2 的核心目标是**提高记忆内容的质量和表达倾向性**，并以此基础构建**三层的「共同记忆撞上」社交引擎**。让用户的每一次表达都带有强烈的个人审美色彩，让「撞上」的那一刻真正产生火花。

### 1.2 设计原则

1. **内容导向**：每一个输入字段的设计都在引导用户回忆「那个独特的瞬间」，而非「随便写点什么」
2. **倾向性即匹配燃料**：表达越具体、越个人化，「撞上」的匹配质量越高
3. **系统搭桥，用户走路**：匹配引擎提供线索和接触点，但不强迫社交
4. **渐进式信任**：信息暴露从少到多，让用户自己决定深入程度

### 3.2 记忆表达引导

#### M-01：一句话记忆句式模板

**设计**：发布 Repo 时，`memoryHook` 字段从自由文本框升级为模板选择器

**四种模板**：

| 模板 | 句式 | 示例 | 表达人格 |
|------|------|------|---------|
| A. 那一刻 | 「____的那一刻，我____」 | 「吉他的第一个音响起的那一刻，我鸡皮疙瘩全起来了」 | 感性派 |
| B. 比喻 | 「这场演出像____」 | 「这场演出像一记重拳打在胸口」 | 比喻狂魔 |
| C. 动作定义 | 「我在这场演出____」 | 「我在这场演出开到了人生中最长的一趟火车」 | 行动派 |
| D. 自由模式 | 纯自由输入 | 「雪纺裙很漂亮，现场不太行」 | 自由人 |

**数据模型变更**：`Repo` 表增加字段

```
model Repo {
  ...
  memoryTemplate  String?   // "A" | "B" | "C" | "D"，用户选择的模板类型
  memoryHook      String    // 已有的字段，保存完整的填空后句子
  ...
}
```

**UI 变更**（RecordPage）：
- 一句话记忆区域改为模板选项卡 + 填空输入框
- 四个模板以标签/卡片形式展示，点击选中
- 选中后显示对应的句子骨架，用户在填空区域输入内容
- 实时显示已填/总字数
- 自由模式保持原有输入框

**伪代码**：
```
// 模板定义
const templates = {
  A: { label: '那一刻', skeleton: '____的那一刻，我____', persona: '感性派' },
  B: { label: '比喻', skeleton: '这场演出像____', persona: '比喻狂魔' },
  C: { label: '动作定义', skeleton: '我在这场演出____', persona: '行动派' },
  D: { label: '自由', skeleton: null, persona: '自由人' },
}
```

**约束**：
- 用户必须选择一个模板（包括自由模式）
- 填空内容长度 ≤ 20 字（含骨架固定文字）

### 3.3 共同记忆撞上系统

#### C-01 ~ C-03：三层级匹配

**匹配引擎逻辑**：

```
撞上检测（触发时机：用户发布 Repo 后 / 浏览条目页时）：

1. Lv1 撞演出：
   - 查询当前 Show 下所有标记了 ATTENDED 的其他用户
   - 输出：计数

2. Lv2 撞记忆（在 Lv1 基础上）：
   - 筛选与当前用户一句话记忆语义相似的其他 Repo
   - 或：对方给当前用户的 memoryHook 点了赞
   - 语义相似度算法：首次用关键词重合度（简单），后续可升级为 embedding
   - 输出：匹配用户列表 + 对方的记忆标签

3. Lv3 撞 vibe（在 Lv2 基础上）：
   - 三维评分差值 ≤ 1 分（每个维度）
   - 审美重合度 = 相似记忆数 / 总同场数
   - 输出：完整用户信息 + 重合度百分比
```

**数据模型**：新增 `Match` 表

```
model Match {
  id              String   @id @default(cuid())
  userAId         String                       // 用户 A
  userBId         String                       // 用户 B
  showId          String                       // 撞上的演出
  matchLevel      Int                          // 1/2/3
  similarityScore Float?                       // 审美重合度百分比 (0-100)
  memoryHookA     String?                      // A 的一句话记忆（用于卡片展示）
  memoryHookB     String?                      // B 的一句话记忆
  matchedAt       DateTime @default(now())
  status          MatchStatus @default(PENDING)

  @@unique([userAId, userBId, showId])         // 同一演出不重复匹配
  @@index([userAId, status])
  @@index([userBId, status])
}

enum MatchStatus {
  PENDING    // 待处理（用户尚未回应）
  CONNECTED  // 已建立连接（双方已互动）
  DISMISSED  // 已忽略
}
```

**API 变更**：

```
# 触发匹配（发布 Repo 后由后端自动触发）
POST /match/trigger/:repoId
  → 系统检测匹配并创建 Match 记录
  Response: { matches: number } // 本次触发的匹配数

# 获取我的匹配
GET /match/my
  → Response: {
      items: [{
        id, matchLevel, similarityScore,
        show: { id, artistName, venueName, showDate },
        otherUser: { id, nickname, avatarUrl },
        otherMemoryHook: string,
        matchedAt,
        status
      }]
    }

# 匹配操作
PATCH /match/:id
  Body: { status: "CONNECTED" | "DISMISSED" }

# 同场盲盒（每日推送）
GET /match/blind-box
  → 返回一条匿名匹配（仅展示对方的记忆和演出，不暴露身份）
  Response: {
    id,           // match id
    matchLevel,
    show: { artistName, venueName, showDate },
    otherMemoryHook: string,   // 匿名展示
  }
```

**UI 变更**：

**Lv1 撞演出 — ShowDetailPage 统计区**：
- 「看过 X 人」增加显示「你去的时候 TA 也在」
- 或：Repo 详情中提到「这场还有 X 位同场者」

**Lv2 撞记忆**：
- 发布 Repo 后的 toast：「你和 TA 对这场演出的感受惊人的相似」
- 点击可查看匹配卡片（半匿名）

**Lv3 撞 vibe**：
- 发现页「与你有感共鸣」区块
- 展示匹配用户及其审美重合度
- 「打个招呼」按钮（预设话术）

#### C-05：同场盲盒

**UI 位置**：发现页（DiscoverPage）新增区块

**每日推送机制**：
- 每日 0 点从所有活跃匹配中随机选一条
- 该匹配的双方用户各收到一次盲盒推送
- 推送后 24h 内有效

**盲盒卡片 UI**：
```
┌────────────────────────────┐
│ 🎸 同场盲盒                 │
│                            │
│ 「跳水的那一刻，             │
│  我失去了所有重量」          │
│                            │
│ —— 来自 XX 演出的某位         │
│                            │
│ [ 我也这么觉得 🤘 ]  [ 滑走 ]  │
└────────────────────────────┘
```

**交互**：
- 「我也这么觉得」→ 揭晓对方身份，Match 状态变更为 CONNECTED
- 「滑走」→ 无反馈，Match 状态保持 PENDING 或变更为 DISMISSED（每日可玩一次）

### 3.4 AI 能力

#### A-01：AI 多声线扩写

**位置**：RecordPage，正文文本区下方

**交互流程**：
1. 用户输入关键词（至少 3 个词）
2. 点击「AI 帮我写」按钮
3. 后端调用 LLM API，返回三种风格版本
4. UI 展示三个版本卡片，用户点击选择
5. 选中版本填入正文文本框

**三种声线**：

| 声线 | 示例 |
|------|------|
| 感官轰炸版 | 「吉他的失真轰鸣穿透胸膛，贝斯的律动砸进骨髓。你被人群推着、挤着、托举着，脚下的地板在震，头顶的灯光在闪。你不知道自己在哪里，只知道这一刻是真的活着。」 |
| 冷峻纪实版 | 「20:30 开场，暖场 MC 介绍时长 3 分钟。第一首前奏一响，全场毫无延迟地进入状态。鼓手的 hi-hat 开镲节奏偏快，整体第一首歌的 tempo 偏激进。」 |
| 段子手版 | 「本来是来养生的，结果被人群挤到舞台第三排。吉他手看了我一眼，我知道他在说'你不行'。所以我现在脖子断了，去他的养生。」 |

**约束**：
- AI 生成是可选功能，不可替代用户手动输入
- AI 失败（API 超时/报错）时不可阻塞发布流程，只隐藏 AI 按钮
- 每次调用限 apiKey 配额，需做频率限制

#### A-02：AI 标签建议

**位置**：自由模式（模板 D）下的一句話记忆输入框

**交互流程**：
1. 用户在正文文本框输入长内容
2. 自动或手动触发 AI 提取关键词
3. AI 返回 1-3 个建议标签
4. 用户点击选择或直接输入

**约束**：
- 仅在自由模式下可用
- 同样不可阻塞发布流程
